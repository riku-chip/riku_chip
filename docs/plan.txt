 Arquitectura GDS: contratos entre gdstk-rs · gds-renderer · riku

 Contexto

 riku es un VCS semántico para chips IC. Hoy solo maneja .sch (xschem) vía un
 driver dedicado. Para soportar archivos .gds (layouts físicos) hay que
 agregar tres cosas:

 1. Extender gdstk-rs con métodos que falten (lectura desde bytes, XOR que
 devuelva polígonos).
 2. Crear gds-renderer — un crate nuevo que convierte polígonos en SVG.
 3. Agregar gds_driver.rs a riku que implementa el trait RikuDriver
 usando los dos crates anteriores.

 La clave de esta arquitectura es que cada capa expone tipos POD (Plain Old
 Data) en su frontera pública. Los contratos son estables — la implementación
 interna puede cambiar, pero las signatures públicas no. Esto permite evolucionar
 cada crate independientemente.

 Las decisiones tomadas con el usuario son:
 - Formato de salida del renderer: SVG (reutiliza svg_annotator existente)
 - Tipo OwnedPolygon: vive en gdstk-rs (es el productor natural)
 - Carga de bytes: Library::from_bytes() agregado a gdstk-rs (evita tempfile)
 - Interactividad en dos fases:
   - Fase 1 (ahora): SVG con grupos por layer + HTML/JS que inyecta riku para
 toggle de layers, toggle de labels, zoom/pan, cambio de colores en vivo.
 Funcional hasta ~50k polígonos (chips chicos/medianos).
   - Fase 2 (futuro): App nativa con egui + wgpu usando el mismo
 gds-renderer como backend geométrico, para chips grandes (millones de
 polígonos, pan/zoom real tipo KLayout).

 Contrato 1: gdstk-rs — API pública para otros crates

 Archivos que se modifican:
 - C:\Users\ariel\Documents\riku_chip\gdstk\rust\src\lib.rs

 Tipos POD ya existentes (no se tocan — son el contrato):
 pub struct Point2D    { pub x: f64, pub y: f64 }                          // Copy
 pub struct BoundingBox { pub min_x, min_y, max_x, max_y: f64 }            // Copy
 pub struct GdsTag     { pub layer: u32, pub datatype: u32 }               // Copy, Hash
 pub struct XorMetrics { pub area: f64, pub region_count: u64,
                         pub bbox: BoundingBox }                           // Copy
 pub enum   ErrorCode  { NoError, BooleanError, ... }                      // Copy

 Tipo nuevo a agregar (vive en lib.rs porque cruza crates sin lifetime):
 /// Polígono dueño de sus datos. Se usa cuando el polígono debe sobrevivir
 /// más allá del Library que lo creó (ej: resultado de XOR, paso a otro crate).
 #[derive(Clone, Debug, PartialEq)]
 pub struct OwnedPolygon {
     pub layer: u32,
     pub datatype: u32,
     pub points: Vec<Point2D>,
 }

 Constructor nuevo — leer desde memoria (evita tempfiles en riku):
 impl Library {
     /// Ya existe: abrir desde archivo
     pub fn open(path: &str) -> Self;

     /// NUEVO: abrir desde bytes en memoria
     /// Implementación: escribe a tempfile internamente, llama a gds_read del
     /// C++ gdstk, cleanup al retornar. Usa std::io::Cursor vía C++ no es
     /// viable porque gdstk solo expone read por path.
     pub fn from_bytes(data: &[u8]) -> Result<Self, Error>;
 }

 Método nuevo — XOR que devuelve los polígonos, no solo métricas:
 impl<'a> Cell<'a> {
     /// Ya existe: solo devuelve área, región count, bbox
     pub fn xor_with(&self, other: &Cell<'_>, layer: u32) -> XorMetrics;

     /// NUEVO: devuelve los polígonos reales del XOR simétrico por layer.
     /// Cada polígono resultante es OwnedPolygon — no tiene lifetime porque
     /// gdstk::boolean() aloja nuevos polígonos en heap que extraemos y
     /// convertimos a Vec<Point2D>.
     pub fn xor_polygons(&self, other: &Cell<'_>, layer: u32)
         -> Vec<OwnedPolygon>;
 }

 Invariantes del contrato:
 - Todos los tipos POD son Copy + Send + Sync.
 - OwnedPolygon es Clone + Send + Sync (no Copy porque tiene Vec).
 - Library es Send pero !Sync (seguro mover entre threads con Arc).
 - Cell<'a>, Polygon<'a> son Copy pero lifetime-bound — nunca se
 exponen en el contrato entre crates. Solo OwnedPolygon cruza fronteras.
 - from_bytes() retorna Result porque falla por I/O; open() no (comportamiento existente).

 Lo que gdstk-rs NUNCA hace:
 - No conoce pixels, colores, imágenes
 - No conoce git
 - No conoce el diff semántico (cosmético vs funcional)

     pub tag: gdstk_rs::GdsTag,   // layer + datatype

 }

 /// Renderiza una celda completa como SVG.
 /// Estructura generada (CONTRATO ESTABLE):
 ///   <svg>
 ///     <g id="layer-metal1" data-layer="68" data-datatype="20">
 ///       <polygon .../>
 ///       ...
 ///     </g>
 ///     <g id="layer-metal1-labels" data-layer="68">
 ///       <text .../>
 ///     </g>
 ///     ... un <g> por layer ...
 ///   </svg>
 /// Los ids 'layer-X' y 'layer-X-labels' son el contrato que riku usa
 /// para inyectar el JS de toggle.
 pub fn render_cell(
     cell: &gdstk_rs::Cell<'_>,
     config: &RenderConfig,
 ) -> RenderOutput;

 /// Renderiza una celda con polígonos del XOR resaltados en rojo.
 /// Usado por riku diff para pintar "lo que cambió".
 /// Agrega un grupo final <g id="layer-highlights"> con los polígonos XOR.
 pub fn render_cell_with_highlights(
     cell: &gdstk_rs::Cell<'_>,
     highlights: &[gdstk_rs::OwnedPolygon],
     config: &RenderConfig,
 ) -> RenderOutput;

 /// El renderer devuelve SVG + la lista de layers presentes, para que riku
 /// pueda generar los checkboxes sin re-parsear el SVG.
 pub struct RenderOutput {
     pub svg: String,
     pub layers: Vec<LayerInfo>,  // metadata que riku usa para armar UI
 }

 Por qué SVG y no PNG:
 - Reutiliza riku/src/core/svg_annotator.rs (ya existe)
 - Integra directo en el HTML side-by-side (coherente con xschem)
 - Permite zoom sin pérdida
 - Para GDS enormes se puede degradar a rasterización interna dentro del SVG si hace falta

 Cómo funciona internamente (no es parte del contrato, puede cambiar):
 1. Calcula viewport desde bbox de la celda
 2. Transforma coords GDS (µm) → coords SVG
 3. Itera layers conocidos del PDK
 4. Para cada layer: itera polígonos, emite <polygon> con color del PDK
 5. Si hay highlights: emite último <g> con los polígonos XOR en rojo/verde
 6. Serializa todo como SVG String

 Lo que gds-renderer NUNCA hace:
 - No lee archivos GDS (recibe Cell<'_> ya parseado)
 - No conoce git
 - No maneja diffs (solo recibe polígonos y los pinta)
 - No abre ventanas ni UI
 - No genera el HTML contenedor ni el JS de toggle — eso es responsabilidad de riku

 Preparado para Fase 2 (app nativa):
 El contrato SVG no cambiará. Para la app nativa se agregará en el futuro una
 función paralela que devuelve mesh/triangulación en vez de SVG, sin romper nada:
 // Fase 2 (NO se implementa ahora, solo se reserva el espacio de diseño):
 pub fn render_cell_native(
     cell: &gdstk_rs::Cell<'_>,
     config: &RenderConfig,
 ) -> NativeRenderOutput;  // polígonos triangulados listos para wgpu

 Archivo nuevo — gds_driver.rs:
 C:\Users\ariel\Documents\riku_chip\riku\src\adapters\gds_driver.rs

 Implementa RikuDriver así:
 - info() → { name: Gds, extensions: [".gds", ".oas"] }
 - diff(a, b, hint):
   a. let lib_a = gdstk_rs::Library::from_bytes(a)?
   b. let lib_b = gdstk_rs::Library::from_bytes(b)?
   c. Toma top-level cells de cada uno
   d. Para cada layer común: calcula XorMetrics + xor_polygons
   e. Convierte a DiffEntry (una entrada por layer con cambios)
   f. Llama gds_renderer::render_cell_with_highlights() para generar
 RenderOutput { svg, layers }
   g. Genera HTML completo (con checkboxes + JS de toggle) envolviendo el SVG
   h. Cachea SVG puro en ~/.cache/riku/ops/{sha256(content)}/render.svg
   i. Cachea HTML interactivo en ~/.cache/riku/ops/{sha256(content)}/render.html
   j. Retorna DriverDiffReport con visual_a/visual_b apuntando a los HTML
 - render(content, hint): misma caché; retorna path al HTML sin highlights
 - normalize(content, _): content.to_vec() (no-op)

 Flujo end-to-end: riku diff <a> <b> layout.gds

 ┌─ riku ─────────────────────────────────────────────┐
 │  cli.rs::run_diff()                                │
 │  ↓                                                 │
 │  analyzer::analyze_diff()                          │
 │  ↓                                                 │
 │  git_service::get_blob(commit_a, "layout.gds")     │
 │  git_service::get_blob(commit_b, "layout.gds")     │
 │  → (Vec<u8>, Vec<u8>)                              │
 │  ↓                                                 │
 │  registry::get_driver_for("layout.gds")            │
 │  → Box<GdsDriver>                                  │
 │  ↓                                                 │
 │  driver.diff(bytes_a, bytes_b, "layout.gds")       │
 └──────────┬─────────────────────────────────────────┘
            │
 ┌─ gds_driver.rs ────────────────────────────────────┐
 │  gdstk_rs::Library::from_bytes(bytes_a)            │
 │  gdstk_rs::Library::from_bytes(bytes_b)            │
 │  ↓                                                 │
 │  para cada layer en union(layers_a, layers_b):     │
 │    metrics = cell_a.xor_with(&cell_b, layer)       │
 │    polys   = cell_a.xor_polygons(&cell_b, layer)   │
 │  ↓                                                 │
 │  gds_renderer::render_cell_with_highlights(        │
 │      &cell_a, &polys, &config) → String (SVG)      │
 │  gds_renderer::render_cell_with_highlights(        │
 │      &cell_b, &polys, &config) → String (SVG)      │
 │  ↓                                                 │
 │  escribe SVGs a ~/.cache/riku/ops/{hash}/          │
 │  ↓                                                 │
 │  construye DriverDiffReport {                      │
 │      file_type: Gds,                               │
 │      changes: DiffEntry por layer,                 │
 │      visual_a: path al SVG A,                      │
 │      visual_b: path al SVG B,                      │
 │  }                                                 │
 └──────────┬─────────────────────────────────────────┘
            │
 ┌─ riku ─────────────────────────────────────────────┐
 │  cli.rs::run_visual()                              │
 │  lee SVGs, los mete en HTML side-by-side           │
 │  abre en browser (mismo flujo que xschem)          │
 └────────────────────────────────────────────────────┘
